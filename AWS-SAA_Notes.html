Regions

A cluster of data centers placed all over the world. Most AWS services are regionally scoped.

If you need to launch a new application, where should you do it?

It depends on the following factors:
1. Compliance to governance & legal requirements: data never leaves a region w/o explicit permission.
2. Proximity to customers: for reduced latency.
3. Available Service within a region: new services and new features arent available in every region.
4. Pricing: this varies region to region. This can be found in the services pricing page.

Availability Zones (AZ's)

Each region has between 3 & 6 AZ's.
Each AZ is one or more discrete data centers with redundant power, networking, & connectivity.
Each zone is geographically seperated from each other so they are isolated from disaster (weather, terrorism, etc)
Each AZ is connected with high bandwidth, ultra-low latency networking. These connections make the AZ's a region.

AWS Points of Presence (Edge Locations)

200+ Points of Presence, 84 cities, in 42 countries.
Delivers content to end users with low latency.

IAM - Identity & Access Management

Root Account - created by default and should not be used or shared.
Users - people within your organization and can be grouped.
Groups - Permissions are assigned to the group and inherited by each user in the group. Can only contain users, not other groups. Users are not required to be in a group. Users can be in multiple groups.
Users or groups can be assigned JSON documents call policies.
Polices - define the permissions of the user using the 'l

IAM Policies Structure

A JSON document containing:
1. Version: policy language version "year-month-day"
2. ID: identifier for policy (Optional)
3. Statement: Required and 1 or more individual statements.

Inside of a statement are:
1. SID: Statement ID (Optional)
2. Effect: Whether the statement allows or denys access.
3. Principal: Account, user, or role to which a policy is applied to.
4. Action: list of resources to which action applies to.
5. Resource: list of resources to which the actions applied to.
6. Condition: for when this policy is in effect.(Optional)

IAM - MFA

Users have access to your account and can change/add/delete. things that they have permissions to.
Protecting Root and IAM users with MFA is crucial
MFA = Password that you know + a sercurity device you own.
The benefit of MFA is that if your account credentials are stolen your account is still safe.
You can require them to change their password after a designated password age.
You can prevent password reuse.

Devices that can be used for MFA:
1. Google Authenticator (phone only)(virtual MFA Device). This can support multiple tokesn on a single device.
2. Authy (multi device)(virtual MFA Device). This can support multiple tokesn on a single device.
3. YubiKey(Universal 2nd Factor (U2F) Security Key). This supports multiple root & IAM users using a single key.
4. Hardware Key Fob MFA Device, provided by Gemalto
5. Hardware Key Fob MFA Device for AWS GovCloud, if you are in the USG, provided by SurePassID

Ways to access AWS

There are three options for accessing AWS.
1. AWS management console with username, password, and MFA.
2. AWS command line interface (CLI): protected by access keys.

Access keys are secret, just like passwords that should not be shared.

EC2 Fundamentals

Knowing EC2 is fundamental to how the cloud works
EC2 = Elastic Cloud Computing = IaaS
EC2 is one of the most popular AWS services.
EC2 mainly consists in the capability of:
1. Renting Virtual Machines (EC2)
2. Storing data on virtual drives, Elastic Block Store (EBS)
3. Distributing load across machines, Elastic Load Balancer (ELB)
4. Scaling the service using an auto scaling group (ASG)

EC2 sizing and configuration options

You can choose:
Operating System (OS): Linux, Windows, or MacOS.
How much computing power and cores (CPU).
How much memory (RAM).
How much storage space being network attached (EBS & EFS) or hardware (EC2 Instance Store).
Network card to include the speed of the card and public IP address.
Firewall rules done in the security group.br> Bootstrap script (configure at first launch) with EC2 user data.

EC2 User Data (Bootstrap)

It's possible to bootstrap our instances using EC2 User Data Scripts.
Bootstrapping means launching of commands when a machine first launches.
The script only runs once at the instance first launch.
The EC2 User Data Script runs with the Root user.
EC2 User data is used to automate boot tasks such as:
1. Installing updates.
2. Installing Software.
3. Download common files from the internet.
4. Anything you can think of.
**The more you have bootstrapped the more the instance has to do at first launch, the longer it could take.

EC2 Instance Types


Take m5.2xlarge for example:
m: instance class
5: generation (AWS improves over time)
2xlarge: size of the instance class.

General Purpose:
Great for a diversity of workloads such as web servers or code repositories.
Balanced between:
-Compute
-Memory
-Networking

Compute Optimized:
Great for compute intensive tasks that require high performance processors:
-Batch processing workloads
-Media transcoding
-High performance web servers
-High performance computing (HPC)
-Scientific modeling and machine learning
-Dedicated gaming servers

Memory Optimized:
Fast performance for workloads that process large data sets in memory.
Use cases:
-High performance relational/non-relational databases.
-Distributed web scale cache stores.
-In-memory databases optimized for business intelligence (BI)
-Applications performing real-time processing of big, unstructured data.

Storage Optimized:
Great for storage-intensive tasks that require high, sequential read and write access to large data sets on local storage
Use cases:
-High frequency online transaction processing (OLTP)
-Relational and NoSQL databases.
-Cach for in-memory databases (like Redis)
-Data warehousing applications
-Distributed file systems.

Security Groups

Security Groups are the fundamental of network security in AWS.
The control how traffic is allowed into or out of our EC2 instances
Security groups only contain ALLOW rules.
Security groups rules can be referenced by IP or by Security Group.

Security groups act as a firewall on the EC2 instances. They regulate:
1. Access to ports
2. Authorized IP ranges both IPv4 & IPv6.
3. Controls inbound network (from other to the instance)
4. Controls outbound network (from instance to other)

Security gropus can be attaches to multiple instances
They are locked down to a ragion/VPC combination.
Lives outside of the instance so if traffic is blocked it will never reach the instance.
It's good practice to use a seperate security group for SSH access.
If your application is not accessible (time out), then it's a security group issue
If your application gives a connection refused error then its an application error.
All inbound traffic is BLOCKED by default
All outbound trafiic is ALLOW by default

Important Ports:
21 - FTP (File Transfer Protocol) - upload files into a file share.
22 - SSH (Secure Shell) - Log into a *NIX instance
22 - SFTP (Secure File Transfer Protocol) - uplaod files via SSH
80 - HTTP - for accessing unsecure webistes.
443 - HTTPS - for accessing secure websites.
3389 - RDP (Remote Desktop Protocol) - log into a Windows instance
